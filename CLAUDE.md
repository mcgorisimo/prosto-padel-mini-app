# «Просто Падел» Mini App

## Назначение репозитория

Этот репозиторий содержит Mini App падел-клуба «Просто Падел».

Актуальный рабочий путь:

`C:\Projects\prosto-padel-mini-app`

Frontend находится непосредственно в корне репозитория. Папка `mini-app/` является старым локальным рабочим путём и не используется.

## Текущее состояние

Текущий frontend:

- React;
- Vite;
- JavaScript;
- Tailwind CSS;
- Playwright;
- временная интеграция с Supabase Auth и PostgreSQL.

Backend на NestJS пока не создан.

Telegram-бот Dynamo Padel, его `knowledge/`, `Procfile` и локальные файлы не являются частью текущей Mini App.

## Утверждённая целевая архитектура

Будущая архитектура:

`React + Vite → REST/SSE → NestJS + Fastify → PostgreSQL → integration worker → CRM adapter → YCLIENTS`

Frontend не должен напрямую обращаться к PostgreSQL, YCLIENTS или другим внешним CRM.

Текущие обращения frontend к Supabase являются временными и должны заменяться поэтапно, без широкого переписывания работающих функций.

Production-инфраструктура, персональные данные, файлы, логи и резервные копии должны размещаться в России.

Архитектура должна оставаться переносимой и не зависеть навсегда от одного облачного поставщика.

## Правила разработки

Перед изменением:

```powershell
cd C:\Projects\prosto-padel-mini-app
git rev-parse --show-toplevel
git status --short
```

Работай только в рамках поставленной задачи.

Не меняй файлы вне задачи и не выполняй широкие рефакторинги без необходимости.

Не переписывай работающие части приложения, если задача этого не требует.

Используй существующие компоненты, helpers, фикстуры и тестовые сценарии.

Сохраняй обратную совместимость, если другое отдельно не согласовано.

Для новых экранов используй существующий UI-kit из `src/components/ui/` и действующие стили проекта.

Расчёт цен должен оставаться централизованным в `src/lib/pricing.js`, пока архитектура цен отдельно не изменена.

Основной мобильный интерфейс рассчитан на ширину до `480px`.

Не обновляй `TASK.md`, если это отдельно не запрошено.

* `git add .`;
* `git add -A`;
* `git add --all`;
* `git commit -a`;
* push;
* merge;

* force-команды.

Добавляй в staging только конкретные файлы текущей задачи.

Перед коммитом проверяй staged-файлы через:


## База данных и SQL

SQL-миграции в репозитории являются источником истины.

Codex не применяет SQL самостоятельно.

Любое изменение базы сначала готовится отдельной миграцией с:

* проверкой состояния до применения;
* основной миграцией;
* проверкой результата;
* способом отмены;
* понятной инструкцией.

SQL вручную применяет пользователь сначала в тестовом окружении.

Production не изменяется без отдельного решения и проверки.

Не придумывай структуру базы без проверки существующих миграций и фактической схемы.

## Секреты

Не добавляй в Git:

* реальные `.env`-файлы;
* Telegram-токены;
* Supabase service-role keys;
* JWT secrets;
* пароли;
* строки подключения;
* Vercel credentials;
* CRM API keys.

Серверные секреты нельзя помещать в frontend-код или переменные Vite, доступные браузеру.

Безопасные шаблоны `.env.example` могут храниться в Git без реальных значений.

## Проверка изменений

Запускай только тесты, связанные с текущей задачей, если полный набор тестов не требуется границей этапа.

Отделяй ошибку тестовой среды от реальной ошибки приложения.

Для ручного запуска:

```powershell
cd C:\Projects\prosto-padel-mini-app

npm.cmd run dev -- --host 127.0.0.1 --port 5173 --strictPort
```

Открыть:

`http://127.0.0.1:5173/`

Основные команды:

```powershell
npm.cmd run build
npm.cmd run test:e2e
```

Не считать функцию завершённой без связанных проверок и понятной инструкции ручной проверки.
