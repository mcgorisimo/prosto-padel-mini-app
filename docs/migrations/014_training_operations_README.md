# Миграция 014: серверные операции индивидуальной тренировки

## Статус проекта

Это только проект SQL в репозитории. Codex его не применяет и к PostgreSQL не
подключается.

Сейчас отдельной тестовой базы нет, а подключённая база является рабочей. Поэтому 014
нельзя применять без отдельного решения пользователя, актуальной резервной копии и
проверенной процедуры восстановления.

Кроме того, применение 014 **сейчас заблокировано**: действующая миграция 011 получает
`matches."pricePerPerson"` из клиентского JSON. Такое значение нельзя считать доверенной
ценой корта. PRECHECK выводит `BLOCKED`, а основная миграция завершается ошибкой
`MIGRATION_014_BLOCKED_TRUSTED_COURT_PRICE_SOURCE_MISSING` до создания объектов.

Миграция 014 зависит от будущей миграции 013. Миграция 013 должна добавить доверенный
серверный расчёт стоимости корта: серверный справочник или другой проверяемый контракт,
но не параметр React/Vite-клиента. До применения и проверки 013 миграция 014 остаётся
заблокированной. После реализации 013 проект 014 нужно повторно аудитировать; одной
правки комментария функции недостаточно.

## Что проектирует 014

После снятия ценового блокера миграция создаёт только четыре функции:

- `training_current_customer_id()` — изолированная граница Supabase Auth;
- `training_request_transition_allowed(text, text)` — PostgreSQL-граф допустимых
  переходов;
- `create_individual_training_request(uuid, jsonb)` — клиентская атомарная операция
  создания;
- `transition_individual_training_request(uuid, text, text, text, text, jsonb)` —
  доверенная серверная операция изменения статуса.

Новые таблицы, колонки, индексы и демонстрационные данные не создаются. Используются
таблицы миграции 012:

- `training_requests`;
- `training_status_history`;
- `training_coach_assignments`.

`matches` остаётся локальной фиксацией корта. `create_booking` и exclusion constraint
обычного бронирования не переписываются.

## Атомарное создание заявки

Клиент передаёт стабильный `client_request_id` и существующий JSON обычной брони без
цены. Функция выполняет в одной PostgreSQL-транзакции:

1. Получает `customer_id` через небольшую Supabase-auth границу.
2. Блокирует advisory-ключ `customer_id + client_request_id`.
3. Ищет существующую заявку с этим ключом и при повторе возвращает её без новой брони.
4. Запрещает `pricePerPerson` и `price_per_person` в клиентском JSON.
5. Проверяет длительность: только `1` или `1.5` часа, то есть 60 или 90 минут.
6. Принудительно задаёт личную бронь: `private/private/isPrivate=true/upcoming`.
7. Вызывает действующий `create_booking`, который атомарно занимает слот.
8. Принимает цену только из результата будущего доверенного серверного контракта.
9. Создаёт `training_requests` со статусом `court_sync_pending` и оплатой `not_due`.
10. Создаёт первую append-only запись истории `request_created`.
11. Создаёт только ожидающее назначение тренера `selection_pending` с `coach_id = NULL`.

Если любой шаг завершается ошибкой, PostgreSQL откатывает и строку `matches`, и заявку,
и историю, и ожидающее назначение. Поэтому клиент не получает локальную бронь без
полной локальной training-записи.

Уникальность `(customer_id, client_request_id)` из 012 остаётся вторым уровнем защиты
после advisory lock. Повтор с тем же ключом возвращает исходную заявку, даже если
клиент повторно нажал кнопку или повторил запрос после сетевой ошибки.

## Почему цена корта блокирует применение

В 012 поле `court_price_amount numeric(12,2)` обязательно. Ноль разрешён как временное
продуктовое решение, но значение всё равно должно прийти из доверенного серверного
источника. Текущий `create_booking` записывает необязательную цену из `p_booking`, то есть
клиент может её выбрать сам.

014 не принимает отдельный параметр цены, удаляет оба клиентских алиаса цены и после
`create_booking` требует непустую серверную цену. На текущем контракте результат будет
`NULL`, функция поднимет `TRAINING_TRUSTED_COURT_PRICE_UNAVAILABLE`, а вся транзакция
откатится. Это сознательное fail-closed поведение.

## Переходы статусов

Серверная функция сначала блокирует заявку `FOR UPDATE`, затем сравнивает фактический
статус с `expected_status`. Несовпадение завершает операцию ошибкой
`TRAINING_STATUS_CHANGED`. Только после проверки графа функция обновляет заявку и в той
же транзакции добавляет историю.

Разрешённый граф 014:

| Исходный статус | Допустимые новые статусы |
| --- | --- |
| `court_sync_pending` | `court_sync_unknown`, `awaiting_coach`, `rejected` |
| `court_sync_unknown` | `awaiting_coach`, `rejected` |
| `awaiting_coach` | `rejected`, `cancel_pending` |
| `confirmed` | `cancel_pending`, `completed` |
| `cancel_pending` | `cancelled` |
| `rejected` | нет |
| `cancelled` | нет |
| `completed` | нет |

Терминальные статусы нельзя реактивировать. Переход `awaiting_coach -> confirmed`
сознательно не включён: в 012 ещё нет реального реестра тренеров, проверенного
назначения и доверенной операции установки цены тренера. Разрешать подтверждение без
этих данных было бы ложным состоянием.

014 не добавляет `change_pending`, `sync_conflict` и другие CRM-состояния. Они должны
появиться вместе с утверждённой CRM-моделью, а не заранее.

## История

Создание заявки и каждый успешный переход добавляют отдельную строку
`training_status_history` в той же транзакции, что и изменение основной записи.
Append-only trigger из 012 продолжает запрещать `UPDATE` и `DELETE` истории.

При создании автор — `customer`, а `actor_id` равен текущему профилю. Доверенные
переходы 014 пока записываются как `system` с `actor_id = NULL`. Полноценный admin actor
потребует отдельной проверенной модели административной роли. Контролируемое
обезличивание автора истории также остаётся отдельной будущей серверной операцией;
внешний ключ `actor_id -> profiles.id ON DELETE RESTRICT` не меняется.

## Права доступа

Прямые `INSERT`, `UPDATE`, `DELETE`, `TRUNCATE`, `REFERENCES` и `TRIGGER` на трёх
таблицах по-прежнему запрещены `PUBLIC`, `anon`, `authenticated` и `service_role`.

- `authenticated` получает `EXECUTE` только на клиентское создание;
- `service_role` получает `EXECUTE` только на доверенный переход статуса;
- `anon` не получает операций;
- внутренние auth/transition helpers не выдаются API-ролям;
- обычное чтение своих данных продолжает работать через RLS миграции 012.

Основные функции имеют `SECURITY DEFINER`, фиксированный `search_path` и
`row_security=off`. Владельцу таблиц разрешено выполнять их внутренний DML, хотя API-роли
не имеют прямых прав. Будущие проверенные `SECURITY DEFINER`-операции смогут использовать
тот же принцип, если их владелец и права будут отдельно проверены.

Supabase-зависимость изолирована в `training_current_customer_id()`, где вызывается
`auth.uid()`, и в grants API-ролям. Граф переходов и транзакционная бизнес-логика
используют стандартный PostgreSQL. При переносе на другой auth-провайдер нужно заменить
auth-boundary и grants, а не таблицы или граф состояний. Supabase Edge Functions для
этого контракта не обязательны.

## Что 014 сознательно не делает

014 не добавляет:

- CRM API, адаптер или внешний ID;
- hold/confirm, webhook или polling;
- Telegram и outbox/очередь;
- конкретного тренера или таблицу тренеров;
- фактическое принятие/отказ/замену тренера;
- перенос брони;
- frontend/React-код;
- тестовые профили, брони или заявки.

Автоматическая CRM-синхронизация, outbox, Telegram, конфликт синхронизации и обратная
проверка CRM должны проектироваться отдельным этапом после выбора поставщика. Реальный
реестр тренеров и переход в `confirmed` также требуют отдельной миграции.

## Неизменность 012 и обычного бронирования

Основная миграция до создания функций снимает детерминированные fingerprints структуры
и строк трёх таблиц 012, а также определения `create_booking`. Перед `COMMIT` она
пересчитывает их и завершает всю транзакцию ошибкой при любом различии.

POSTCHECK повторно сверяет полный manifest таблиц с `schema_md5`, записанным самой 012:
колонки, defaults, constraints, foreign keys, индексы, политики, triggers, owner, ACL и
RLS flags. Он также сравнивает `create_booking`, booking exclusion constraint, политики и
grants `matches` с legacy-fingerprints 012.

014 не устраняет остаточный риск legacy-изменения строки `matches` после создания
training-заявки. Перенос активной брони позже должен выполняться одной серверной
транзакцией, синхронно обновляя снимки заявки и историю. До появления такой операции
клиентский сценарий нельзя считать полным.

## Проверки и их ограничения

PRECHECK работает в read-only транзакции и проверяет:

- роли API;
- наличие и ключевой состав 012;
- RLS и отсутствие эффективного прямого DML;
- idempotency/duration constraints;
- защитные функции 012;
- контракт `create_booking`;
- отсутствие объектов 014;
- наличие доверенного источника цены.

POSTCHECK также read-only и не создаёт тестовых данных. Он проверяет:

- точные сигнатуры, return types, security mode и `search_path` функций;
- структурные признаки блокировок, идемпотентности, истории и переходов;
- сохранённые definition/owner/ACL fingerprints функций;
- эффективные права API-ролей;
- неизменность полного manifest 012 и legacy booking-объектов;
- append-only историю и запрет физического удаления заявки.

Структурный POSTCHECK не заменяет поведенческие тесты. Гонки двух запросов, повтор после
timeout, rollback после ошибки цены, конфликт слота и отрицательные переходы нужно
проверять на disposable/test PostgreSQL. Такой базы сейчас нет, поэтому эти тесты не
выполнялись и не должны выполняться на рабочей базе.

## Безопасный порядок ручного запуска

Сейчас SQL запускать нельзя: ожидаемый результат ценовой проверки — `BLOCKED`.

После отдельного решения по тарифу, повторного аудита, резервной копии и проверки
восстановления порядок будет таким:

1. Запустить `014_training_operations_PRECHECK.sql` и прислать полный вывод.
2. Убедиться, что ценовой статус `READY`, а остальные проверки не дали ошибки.
3. Отдельно получить решение пользователя на применение основной миграции.
4. Запустить `014_training_operations.sql` и прислать полный вывод.
5. Запустить `014_training_operations_POSTCHECK.sql` и прислать полный вывод.

Нельзя продолжать после ошибки или неожиданного определения объекта. Codex не должен
запускать эти файлы вместо пользователя.

## Rollback

Rollback удаляет только четыре функции 014 и никогда не удаляет таблицы или данные 012,
`matches`, `profiles`, `create_booking` либо exclusion constraint.

Перед первым `DROP` он проверяет:

- отсутствие неожиданных overloads и других объектов с marker 014;
- definition, owner и ACL fingerprint каждой найденной функции;
- право текущей роли удалить объект;
- отсутствие зависимостей более поздней миграции.

Если часть функций отсутствует, подтверждённая оставшаяся часть может быть удалена.
Если функция изменена или от неё зависит новый объект, rollback останавливается до
любого удаления. Сначала нужно отменить более позднюю миграцию, затем повторить rollback
014. Все `DROP FUNCTION` выполняются с `RESTRICT` внутри одной транзакции.

Так как rollback 014 не удаляет строки, отдельный токен уничтожения данных ему не нужен.
