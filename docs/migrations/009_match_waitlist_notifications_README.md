# 009 — очередь ожидания и уведомления

Этот комплект добавляет серверную очередь ожидания к публичным матчам и общую ленту уведомлений. Он рассчитан на ручную установку поверх уже установленной миграции `008_match_invitations_stage1`.

## Что добавляется

- `public.match_waitlist` хранит историю ожидания. Только строка со статусом `waiting` занимает место в очереди.
- `public.notifications` хранит безопасные уведомления типов `match_invitation` и `waitlist_promoted`.
- RPC `get_my_match_waitlist_position(uuid)` возвращает номер текущего пользователя в поле `queue_position`; имя `position` намеренно не используется, чтобы не конфликтовать с синтаксисом PostgreSQL.
- Ожидающий игрок не попадает в `participants` или `filledSlots`. При освобождении места внутренняя функция блокирует строку матча, берет первого подходящего игрока по `joined_at, id`, повторно проверяет условия и атомарно добавляет его в оба поля.
- Неподходящий к моменту продвижения игрок получает статус `skipped`; проверка продолжается со следующего по FIFO.
- Продвижение вызывается после `leave_match`, отказа от приглашения, отмены приглашения организатором/администратором и нового RPC `remove_match_participant`.
- Создание приглашения и продвижение очереди создают уведомления в той же транзакции. Уникальный `dedupe_key` не дает создать дубль при повторной попытке.

Существующие приглашения, созданные до `009`, намеренно не получают задним числом новые уведомления: миграция не переписывает пользовательские данные. Они по-прежнему доступны через установленный в `008` RPC входящих приглашений.

## Что остается совместимым

Сигнатуры и типы ответов существующих `join_match`, `leave_match`, `create_match_invitation`, `decline_match_invitation` и `cancel_match_invitation` не меняются. Текущий frontend может продолжить работу без немедленного обновления.

Для `leave_match` не требуется маркер миграции `006`. PRECHECK проверяет фактически установленную функцию: `leave_match(uuid) returns public.matches`, `SECURITY DEFINER`, фиксированный `search_path = public, pg_temp`, блокировку строки матча, запрет выхода организатора, защиту оплаченного слота и удаление только текущего пользователя из `participants` и `filledSlots`.

Перед заменой основная миграция сохраняет точный результат `pg_get_functiondef()` и метаданные фактической функции во внутренней таблице `prosto_padel_internal.migration_009_function_state`. В самой миграции для неё включается RLS без пользовательских политик, а все права на таблицу и внутреннюю схему явно отзываются у `PUBLIC`, `anon` и `authenticated`. Поэтому клиентское приложение не может читать или менять снимок. Владельцем остаётся роль, запустившая миграцию из SQL Editor: она может сохранить снимок, выполнить POSTCHECK и при необходимости прочитать его во время ROLLBACK.

Важно: прямой `UPDATE public.matches`, который старый frontend использует для изменения `participants` и `filledSlots`, на этом этапе намеренно не запрещен. Такой UPDATE может обойти приглашения, очередь и автоматическое продвижение. Полная серверная защита появится только после перевода frontend на RPC и отдельного отключения legacy-доступа.

## Supabase-зависимости

- `auth.uid()` — текущий пользователь. При переезде его нужно заменить адаптером идентификации с тем же смыслом.
- Роли `authenticated` и `anon` — только API-права; их можно заменить ролями нового API.
- RLS защищает строки очереди и уведомлений. Это обычный PostgreSQL; Supabase дает JWT-контекст для `auth.uid()`.
- RPC вызываются через PostgREST/Supabase, но являются обычными функциями PostgreSQL.
- Realtime миграция не включает и для корректности очереди не требует. Его можно подключить позже только для мгновенного обновления интерфейса.

## Ручное применение к рабочей базе

Отдельной staging-базы нет, поэтому выберите время низкой активности и заранее сделайте резервную копию или убедитесь, что доступно восстановление на момент до миграции.

1. Откройте SQL Editor текущей базы.
2. Выполните целиком `009_match_waitlist_notifications_PRECHECK.sql`.
3. Остановитесь, если запрос завершился ошибкой или `precheck_ok` не равен `true`. Ничего из основной миграции не запускайте.
4. Сохраните полный JSON PRECHECK, включая фактическое определение `leave_match` из `known_functions`, и актуальную резервную копию базы.
5. Убедитесь, что в приложении нет активной ручной операции изменения состава матча. Основная миграция не переписывает `matches`, но на короткое время заменяет определения RPC.
6. Выполните целиком `009_match_waitlist_notifications.sql` одним запуском. Не запускайте отдельные фрагменты.
7. Если SQL Editor показал ошибку, транзакция откатится целиком. Не повторяйте запуск до разбора текста ошибки.
8. Сразу выполните целиком `009_match_waitlist_notifications_POSTCHECK.sql`. Он создает тестовые строки только внутри транзакции и заканчивается `ROLLBACK`.
9. Убедитесь, что `postcheck_ok = true`, `internal_state_table_private = true`, `internal_schema_hidden = true`, `existing_data_unchanged = true` и в `note` нет ошибок.
10. Минимально проверьте в приложении: обычное вступление в неполный публичный матч; приглашение в частный матч; самостоятельный выход неоплаченного участника. Frontend очереди и уведомлений появится отдельным этапом.

POSTCHECK детерминирует время двух тестовых записей очереди внутри своей откатываемой транзакции. Это необходимо, потому что `now()` стабилен в пределах транзакции: без фиксации времени обе записи получили бы одинаковый `joined_at`, и тест случайно упорядочил бы их по UUID вместо порядка тестовых вызовов. Проверяется полный сценарий: позиции 1 и 2, выход первого пользователя, новая позиция 1 у второго и отсутствие активной позиции у вышедшего.

При проверке сохраненного `pg_get_functiondef()` допустимы обе эквивалентные формы возвращаемого типа — `RETURNS matches` и `RETURNS public.matches`. PostgreSQL может опустить схему для типа, видимого через `search_path`; пригодность точного сохраненного определения для ROLLBACK от этого не меняется.

## Когда немедленно остановиться

- PRECHECK не вернул `precheck_ok = true`.
- Установлена не та версия `008` или фактический `leave_match` не прошёл проверку совместимой атомарной логики.
- Найдены конфликтующие таблицы, схема или RPC.
- Есть `filledSlots` не-массив, больше четырех слотов или некорректные приглашения.
- Основная миграция превысила `lock_timeout`/`statement_timeout` или вернула любую ошибку.
- POSTCHECK не вернул `postcheck_ok = true`.

## Отмена

`009_match_waitlist_notifications_ROLLBACK.sql` восстанавливает RPC приглашений/вступления из `008`, а `leave_match` — из точного снимка функции, сделанного основной миграцией непосредственно перед заменой. Поэтому будет возвращена фактическая версия рабочей базы вместе с её `search_path`, владельцем, ACL и комментарием, а не предполагаемая версия `006`. Затем удаляются только объекты `009`. Для безопасности ROLLBACK откажется работать, если в очереди или уведомлениях уже появились постоянные строки: их удаление означало бы потерю пользовательских данных.

Сразу после установки, пока новые таблицы пусты, ROLLBACK можно выполнить целиком. После него повторно запустите PRECHECK: ожидаемо `no_009_conflicts` снова будет `true`, а наличие `009` — `false`.
