# Миграция 011: сохранение цены бронирования

## Что исправляет комплект

Новые бронирования уже рассчитывают стоимость участия одного игрока во
frontend. Раньше `create_booking` не сохранял это значение в существующую
колонку `matches."pricePerPerson"`. Поэтому приглашение получало `NULL`, а
старый frontend превращал его в `0 ₽`.

Миграция 011 сохраняет переданный снимок цены. Тарифы и старые строки не
пересчитываются и не изменяются.

## Файлы

- `011_create_booking_price_snapshot_PRECHECK.sql` — только проверяет совместимость.
- `011_create_booking_price_snapshot.sql` — обновляет `create_booking(jsonb)`.
- `011_create_booking_price_snapshot_POSTCHECK.sql` — проверяет публичную,
  частную и совместимую старую форму вызова; все тестовые записи откатываются.
- `011_create_booking_price_snapshot_ROLLBACK.sql` — восстанавливает точную
  функцию, сохранённую перед установкой 011.
- этот README — порядок ручных действий.

## Что меняется

Сигнатура остаётся прежней:

```sql
public.create_booking(p_booking jsonb) returns public.matches
```

Новый frontend передаёт `pricePerPerson`. Функция принимает также
`price_per_person`, чтобы не зависеть от регистра ключа интеграции.

- Положительная корректная цена сохраняется без пересчёта.
- Отсутствующий ключ по-прежнему допустим и сохраняет `NULL` — это сохраняет
  обратную совместимость старых клиентов.
- Ноль, отрицательная, `NaN` или чрезмерная цена отклоняются сообщением
  `BOOKING_INVALID_PRICE_PER_PERSON`.
- Существующие строки не обновляются и не получают придуманную цену.

## Как установить вручную

1. Откройте Supabase Dashboard нужного проекта.
2. Перейдите в SQL Editor.
3. Выполните целиком `011_create_booking_price_snapshot_PRECHECK.sql`.
4. Убедитесь, что результат содержит `precheck_ok = true`.
5. Выполните целиком `011_create_booking_price_snapshot.sql`.
6. Выполните целиком `011_create_booking_price_snapshot_POSTCHECK.sql`.
7. Убедитесь, что результат содержит `postcheck_ok = true`.
8. После этого опубликуйте frontend, который передаёт `pricePerPerson`.

POSTCHECK начинает транзакцию и завершает её `ROLLBACK`, поэтому созданные им
брони не остаются в базе.

POSTCHECK можно безопасно запускать повторно. Проверка служебного снимка
сравнивает ACL по фактическому набору прав, а не по порядку элементов системного
массива `aclitem[]`. В результате также выводится `rollback_checks`: отдельные
флаги показывают RLS, отсутствие политик и клиентских прав, закрытость внутренней
схемы, доступ владельца для административного отката и целостность сохранённой
версии `create_booking`.

ROLLBACK использует такое же семантическое сравнение ACL. Порядок записей
`authenticated` и `service_role` не влияет на права и больше не блокирует
восстановление сохранённого определения функции.

## Откат

Если до отката функцию `create_booking` не заменяли другой миграцией, выполните
`011_create_booking_price_snapshot_ROLLBACK.sql`. Он восстанавливает точное
определение, владельца, права, `search_path` и комментарий функции, сохранённые
до установки 011, а затем удаляет только служебный снимок 011.

Откат не изменяет созданные после установки матчи и их сохранённые цены.

## Что проверить после установки

- Создать публичную игру через экран бронирования и отправить приглашение.
- Сравнить сумму в приглашении и деталях матча — она должна совпадать.
- Создать частную бронь и убедиться, что `pricePerPerson` тоже сохранён.
- Открыть старую игру без цены: вместо `0 ₽` или нового расчёта должно быть
  написано `Стоимость уточняется`.
