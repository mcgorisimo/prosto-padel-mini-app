# Миграция 010: безопасный список ожидания публичного матча

Этот комплект добавляет один новый RPC `get_match_waitlist(p_match_id uuid)`. Он ничего не меняет в матчах, очереди, профилях или логике продвижения.

## Кто увидит очередь

Список доступен любому вошедшему пользователю, если он открыл публичный матч. Это может быть организатор, участник, ожидающий игрок или другой авторизованный пользователь.

Для частного матча RPC возвращает ошибку `WAITLIST_PUBLIC_MATCH_ONLY`. Неавторизованный запрос не имеет права запуска RPC; внутри функции дополнительно проверяется `auth.uid()`.

## Что возвращается

Для каждой строки со статусом `waiting` возвращаются только:

- `waitlist_id`;
- `user_id`;
- `queue_position`;
- `first_name`;
- `last_name` — не полная фамилия, а только первая буква с точкой;
- `photo_url` — поле реально существует в текущей таблице `profiles`, пустая строка превращается в `null`;
- `rating`;
- `joined_at`;
- `is_current_user`.

Телефон, email, username, роль, дата рождения, пол, язык, предпочитаемая сторона и признак верификации не возвращаются.

Полный список не открывается прямым `SELECT` к `match_waitlist`. Миграция не добавляет таблице политики или права. Существующая защита 009, при которой пользователь напрямую видит только собственную строку, остаётся без изменений. Полный безопасный список отдаёт только RPC.

## Почему порядок совпадает с продвижением

И RPC, и внутренняя функция автоматического продвижения 009 сортируют ожидающих одинаково: сначала по `joined_at`, а при одинаковом времени — по `id`. Поэтому позиция в ответе соответствует серверному FIFO на том же снимке данных.

## Как установить вручную

Перед началом сделайте резервную копию базы или убедитесь, что можете восстановить её состояние.

1. Откройте SQL Editor рабочего проекта Supabase.
2. Целиком выполните `010_match_waitlist_public_view_PRECHECK.sql`.
3. Найдите в JSON поле `precheck_ok`. Продолжайте только если оно равно `true`.
4. Сохраните полный JSON PRECHECK. В нём указаны найденные реальные поля имени, рейтинга и аватара.
5. Целиком одним запуском выполните `010_match_waitlist_public_view.sql`.
6. Сразу целиком выполните `010_match_waitlist_public_view_POSTCHECK.sql`.
7. Убедитесь, что `postcheck_ok = true`, `behavior.test_executed = true`, `existing_data_unchanged = true`, а `note` пустой.
8. Если любой шаг вернул ошибку или итоговое значение не равно `true`, остановитесь и сохраните полный результат для разбора.

PRECHECK только читает каталоги. POSTCHECK создаёт тестовые матчи и строки очереди внутри транзакции, а в конце всегда выполняет `ROLLBACK`, поэтому тестовые записи не сохраняются.

## Как отменить

`010_match_waitlist_public_view_ROLLBACK.sql` удаляет только RPC миграции 010. Таблицы и пользовательские данные он не меняет. Перед удалением он проверяет метку функции и откажется работать, если функция уже заменена другой версией.

После ROLLBACK повторный PRECHECK ожидаемо покажет, что конфликтующей функции больше нет, а миграция 009 по-прежнему установлена.

## Когда остановиться

- `precheck_ok` не равен `true`;
- не найдена установленная миграция 009;
- реальные поля `profiles` отличаются от `first_name`, `last_name`, `photo_url`, `rating`;
- уже существует функция с именем `get_match_waitlist`;
- основная миграция вернула ошибку;
- `postcheck_ok` не равен `true` или в `note` есть текст.

Frontend и миграции 000–009 этим комплектом не изменяются.
