# Управленческий план Mini App «Просто Падел»

Этот документ — основной план развития проекта. Он фиксирует подтверждённую архитектуру, порядок работ, критерии готовности и ограничения. Этап считается завершённым только после подтверждённых проверок.

## 1. Цель проекта

Запустить российскую Telegram Mini App падел-клуба «Просто Падел» с профилем игрока, бронированием и оплатой, интеграцией с YCLIENTS, публичными и приватными матчами, существующим рейтингом, первичной оценкой уровня и Tournament MVP.

## 2. Подтверждённая архитектура

Текущая работающая схема:

> Telegram → React/Vite frontend на Vercel → Supabase Auth и PostgreSQL

Целевая схема:

> Telegram → React/Vite frontend на российских серверах → NestJS REST/SSE backend → PostgreSQL → transactional outbox и worker → платёжная система и YCLIENTS

Архитектурные решения:

- frontend: React + Vite;
- backend: NestJS + Fastify;
- PostgreSQL-first: PostgreSQL хранит основное состояние и транзакционную бизнес-логику;
- REST используется для команд и запросов;
- SSE используется для серверных событий;
- transactional outbox гарантирует надёжную постановку внешних действий и событий;
- отдельный worker обрабатывает outbox и фоновые задачи;
- CRM adapter изолирует интеграцию с YCLIENTS;
- серверы и персональные данные размещаются в России;
- система не должна постоянно зависеть от одного поставщика;
- frontend не обращается напрямую к PostgreSQL или YCLIENTS;
- SQL-миграции в репозитории являются источником истины.

### Решение по существующей Supabase-схеме

Уже созданная предметная модель не проектируется повторно. Нужно максимально переиспользовать:

- `profiles`;
- `matches`;
- `messages`;
- рейтинг;
- приглашения;
- waitlist;
- уведомления;
- функции;
- триггеры;
- индексы;
- ограничения;
- транзакционную PostgreSQL-логику.

Нужно адаптировать или удалить зависимости от:

- `auth.users`;
- `auth.uid()`;
- Supabase RLS;
- ролей `anon`, `authenticated`, `service_role`;
- прямого доступа frontend к базе;
- Supabase RPC как публичного API;
- Supabase Realtime.

В целевой архитектуре используются:

- `backend_auth.accounts`;
- серверная сессия NestJS;
- repositories;
- REST;
- SSE.

Старые строки пользователей, профилей, матчей и сообщений автоматически не переносятся.

## 3. Текущее состояние

Подтверждённо завершены:

- аудит старой платформы;
- проектирование российской архитектуры;
- Git-нормализация;
- NestJS backend foundation;
- авторизационные контракты;
- Telegram `initData` verifier;
- state machines;
- миграция 015, содержащая 14 таблиц авторизации;

Эти результаты являются основой следующих этапов, но не означают готовность всей Mini App к запуску.

## 4. Активные параллельные направления

### 07.5.3 — Telegram test-контур на Selectel

**Цель:** подготовить изолированный путь:

> отдельный test-бот → HTTPS test-домен → frontend на Selectel → временно Supabase

Основной Telegram-бот не изменяется.

**Результат:** тестовая Mini App открывается через отдельного бота и российский HTTPS-контур, сохраняя временную связь с Supabase.

**Критерий готовности:** вход и основной пользовательский сценарий проверены в test-боте на iPhone, Android и Telegram WebView; основной бот не затронут.

### 07.6 — PostgreSQL repositories для auth

**Цель:** подключить NestJS ко всем 14 таблицам `backend_auth`.

**Результат:** авторизационный слой работает через repositories и не зависит от прямого доступа frontend к базе.

**Критерий готовности:** все 14 таблиц покрыты repositories, интеграционные тесты проходят на test-базе, ошибки и транзакции обработаны.

### 08.0 — Аудит существующей Supabase-схемы

**Цель:** определить, что переносится без изменений, что адаптируется, что переносится в NestJS, а что удаляется.

**Результат:** согласована карта объектов схемы и зависимостей с отдельным решением для таблиц, функций, триггеров, индексов, ограничений, RLS, RPC и Realtime.

**Критерий готовности:** для каждого объекта зафиксирована целевая судьба и зависимости; SQL не применяется в рамках аудита.

### YCLIENTS readiness

**Цель:** заранее подготовить интеграцию без изменения production.

**Результат:** собраны доступы и токены, подтверждены API methods, mapping ресурсов, read-only проверки и тестовый план.

**Критерий готовности:** тестовые запросы выполняются в разрешённом контуре, mapping согласован, известны ограничения API и сценарии ошибок.

## 5. Последовательность этапов

### 07.5.3 — Telegram test-контур

**Цель:** создать безопасный независимый test-путь через Selectel.  
**Результат:** отдельный бот, HTTPS test-домен и frontend на Selectel временно работают с Supabase.  
**Критерий готовности:** тестовый сценарий подтверждён на целевых клиентах, основной бот не изменён.

### 07.6 — PostgreSQL repositories для auth

**Цель:** реализовать серверный доступ ко всем 14 таблицам `backend_auth`.  
**Результат:** NestJS читает и изменяет auth-данные только через repositories.  
**Критерий готовности:** интеграционные тесты repositories проходят, транзакционные границы подтверждены.

### 07.7 — Telegram login orchestration

**Цель:** связать проверку Telegram `initData`, поиск или создание account и выдачу результата входа.  
**Результат:** единый серверный сценарий Telegram login.  
**Критерий готовности:** успешный, повторный, просроченный и поддельный вход покрыты тестами.

### 07.8 — Серверные сессии и REST

**Цель:** реализовать безопасную серверную сессию NestJS и REST-контракты авторизации.  
**Результат:** frontend получает и завершает сессию через backend.  
**Критерий готовности:** создание, проверка, обновление и отзыв сессии протестированы.

### 07.9 — Реальный Telegram login через Selectel

**Цель:** проверить полный вход через test-бот и российский контур.  
**Результат:** пользователь Telegram входит через frontend и NestJS в test-среде.  
**Критерий готовности:** end-to-end сценарий проходит на iPhone, Android и Telegram WebView.

### 08.0 — Аудит существующей Supabase-схемы

**Цель:** составить точный план переиспользования предметной модели.  
**Результат:** объекты разделены на переносимые, адаптируемые, переносимые в NestJS и удаляемые.  
**Критерий готовности:** карта схемы проверена по SQL-миграциям — источнику истины.

### 08.1 — Профили и рейтинг

**Цель:** перенести существующую логику профилей и рейтинга без ненужного переписывания.  
**Результат:** профиль и рейтинг доступны через NestJS repositories и REST.  
**Критерий готовности:** основные операции и расчёты дают ожидаемые результаты в test-контуре.

### 08.2 — Матчи и сообщения

**Цель:** перенести существующую логику матчей и сообщений на серверный API.  
**Результат:** создание, просмотр, присоединение, слоты и чат работают через REST/SSE.  
**Критерий готовности:** главный сценарий матча проходит end-to-end, а конкурентное занятие слота безопасно.

### 08.3 — Приглашения, waitlist и уведомления

**Цель:** перенести эти функции, только если они входят в ближайший MVP.  
**Результат:** согласованный минимальный набор работает через backend.  
**Критерий готовности:** границы MVP подтверждены, выбранные сценарии покрыты тестами.

### 09 — Перевод frontend на NestJS REST/SSE

**Цель:** убрать прямой Supabase-доступ из frontend.  
**Результат:** frontend использует только NestJS REST/SSE; Supabase RPC и Realtime не являются публичным API.  
**Критерий готовности:** сетевые проверки не находят прямых обращений frontend к базе или YCLIENTS, пользовательские сценарии проходят.

### 10 — Booking Core

**Цель:** реализовать клуб, корты, доступность, временный hold, бронь, отмену, перенос и защиту от двойной продажи.  
**Результат:** полный серверный жизненный цикл брони с атомарными переходами.  
**Критерий готовности:** конкурентные тесты подтверждают, что один слот нельзя продать дважды.

### 11 — Payment Core

**Цель:** реализовать `payment attempt`, `payment`, `refund`, `receipt`, webhook, reconciliation и `unknown result`.  
**Результат:** оплата и возврат связаны с бронью и безопасны при повторах.  
**Критерий готовности:** повторные webhooks не дублируют операции, чеки и возвраты проверены, неизвестный результат не показывается как успех.

### 12 — YCLIENTS integration

**Цель:** реализовать resource mapping, `company_id`, `staff_id`, CRM commands, CRM logs, idempotency, `unknown result`, reconciliation и webhooks.  
**Результат:** взаимодействие с YCLIENTS изолировано CRM adapter и выполняется через надёжные фоновые команды.  
**Критерий готовности:** успешные, повторные, ошибочные и неопределённые ответы проверены без частичного состояния.

### 13 — Полный сценарий бронирования

**Цель:** соединить цепочку «выбор → hold → оплата → YCLIENTS → подтверждение».  
**Результат:** пользователь получает подтверждённую бронь либо понятный безопасный результат ошибки.  
**Критерий готовности:** end-to-end тесты проходят для успеха, отказа, таймаута, повтора и восстановления.

### 14 — Публичные и приватные матчи

**Результат:** публичные матчи видны в ленте, приватные брони в публичную ленту не попадают.  
**Критерий готовности:** видимость, доступ, слоты и сообщения проверены end-to-end.

### 15 — Подключение рейтинга и оценки уровня

**Цель:** подключить существующий рейтинг и первичную оценку уровня.

**Критерий готовности:** расчёты сверены с существующей логикой и покрыты тестовыми примерами.

## 7. Критерии качества

- операции атомарны;
- повторный запрос безопасен;
- слот нельзя продать дважды;
- платёж не применяется дважды;
- неизвестный результат не показывается как успех;
- ошибки YCLIENTS не оставляют частичное состояние;
- есть reconciliation;
- есть audit logs;
- пользователь видит нейтральные и понятные ошибки;
- PostgreSQL и NestJS не открыты напрямую в интернет;
- пользовательские сценарии тестируются на iPhone, Android и Telegram WebView.

## 8. Правила миграций

Для каждого изменения схемы соблюдается порядок:

1. Проектирование.
2. Отдельная SQL-миграция.
3. `PRECHECK`.
4. Применение пользователем вручную сначала на test.
5. `POSTCHECK`.
6. Проверка rollback.
7. Проверка backup/restore.
8. Отдельное решение перед любым изменением production.

Codex готовит и проверяет материалы, но не применяет SQL. SQL-миграции в репозитории остаются источником истины.

## 9. Что не делать

- не переносить старую схему одной командой;
- не переписывать работающую бизнес-логику без необходимости;
- не переносить старые данные автоматически;
- не подключать frontend напрямую к PostgreSQL;
- не подключать frontend напрямую к YCLIENTS;
- не открывать порты 3000 и 5432;
- не переключать основной бот до полной приёмки;
- не отключать Supabase и Vercel преждевременно;
- не обновлять `TASK.md`.
