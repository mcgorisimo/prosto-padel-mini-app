# AGENTS.md

## Продуктовый контекст

Проект «Просто Падел» — это Telegram Mini App для падел-клуба, который должен постепенно вырасти в полноценный продукт для игроков клуба.

Главная цель продукта: создать удобное приложение для игроков падел-клуба «Просто Падел», где человек может бронировать корты, записываться на тренировки, смотреть турниры и клубные события, создавать открытые матчи, присоединяться к матчам других игроков, видеть свой профиль, уровень и историю игр, а в будущем оплачивать бронирования, абонементы, тренировки и турниры.

Референс по логике: Playtomic. Мы не копируем Playtomic целиком. Мы делаем MVP для одного клуба — «Просто Падел».

Отдельное приложение рассматривается только после проверки спроса и активности пользователей внутри Telegram Mini App.

## Как относиться к проекту

Это не обычный сайт и не лендинг. Это будущий продукт для паделистов.

Каждое решение должно приближать приложение к ключевым сценариям:

- игрок бронирует корт;
- игрок записывается на тренировку;
- игрок создает открытый матч;
- игрок присоединяется к матчу;
- игрок видит свои игры, уровень и прогресс;
- клуб управляет расписанием, событиями и заявками.

Если изменение не помогает этим сценариям, его нужно отложить или обосновать.

## Правила работы с кодом

- Не переписывать весь проект без явной необходимости.
- Не менять дизайн без причины.
- Не удалять рабочий код, если пользователь прямо не попросил.
- Не делать хаотичные рефакторинги.
- Не менять архитектуру ради красоты.
- Не смешивать продуктовые изменения, визуальные правки и технический рефакторинг в один большой коммит.
- Делать изменения маленькими шагами.
- После каждой правки объяснять, что именно изменилось и зачем.
- Уважать существующие компоненты, стили и пользовательские сценарии.
- Если в проекте уже есть подходящий паттерн, использовать его.
- Если есть риск сломать рабочий сценарий, сначала проверить его вручную или сборкой.

## Приоритет QA

Приоритет проекта сейчас — стабильность MVP и проверка основных сценариев. Тестирование важнее добавления новых функций.

Перед новой функциональностью сначала проверить:

- работает ли авторизация;
- открывается ли главная;
- работает ли навигация нижнего меню;
- можно ли выбрать слот бронирования;
- можно ли создать личную бронь;
- можно ли создать открытый матч;
- появляется ли открытый матч в ленте;
- можно ли открыть детали матча;
- можно ли присоединиться к матчу;
- корректно ли показываются профиль, рейтинг и история игр;
- понятны ли пользователю статусы брони, матча и тренировки.

Если найден дефект в базовом сценарии, исправлять его раньше, чем добавлять новую функцию.

## Продуктовые принципы MVP

- Сценарии должны быть честными: если оплаты еще нет, интерфейс не должен создавать ощущение, что деньги реально списываются.
- Пользователь должен понимать, что произошло после действия: бронь создана, заявка отправлена, матч опубликован, место занято или действие невозможно.
- Для Telegram Mini App важны короткие экраны, быстрые действия и понятная обратная связь.
- Главный экран должен вести к ключевым действиям: бронь, тренировки, матчи, профиль/рейтинг.
- Матчи и бронирования должны быть связаны между собой: публичный матч занимает слот в расписании, приватная бронь не должна попадать в открытую ленту.
- Админские функции должны добавляться только тогда, когда пользовательские сценарии достаточно стабильны.

## Критическая безопасность Git

Git-корень проекта: C:\Projects\prosto-padel-mini-app.
Перед любыми Git-операциями обязательно проверять его командой git rev-parse --show-toplevel.

В этом репозитории строго запрещено выполнять массовое добавление файлов:

- `git add .`
- `git add -A`
- `git add --all`
- `git commit -a`

Также запрещено без отдельного явного разрешения пользователя выполнять:

- `git clean -fd`
- `git reset --hard`
- `git push`
- `git merge`
- `git rebase`

Перед любым коммитом обязательно выполнить:

1. `git rev-parse --show-toplevel`
2. `git status --short`

В staging разрешено добавлять только заранее проверенные файлы явными путями, например:

`git add src/lib/booking.js tests/e2e/mini-app.spec.js`

Никогда не использовать `git add .`.

После добавления файлов обязательно выполнить:

`git diff --cached --name-only`

В выводе должны быть только файлы текущей задачи внутри:

C:\Projects\prosto-padel-mini-app

Если появился любой путь из соседнего проекта или путь вне текущей задачи:

- остановиться;
- не создавать commit;
- не удалять файлы;
- сообщить пользователю полный список лишних путей.

Push выполняется только по отдельному явному запросу пользователя.
